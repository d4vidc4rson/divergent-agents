# Wrong Problem Detector Protocol

Before generating solutions, run five checks on the problem itself. When every solution to a problem feels mediocre, the problem is usually mislabeled -- you're optimizing the wrong variable. Five minutes confirming the right problem saves hours of brilliant answers to the wrong one.

---

## Check 1: The Symptom Test

**Ask: "Is this the problem, or is this what the problem looks like?"**

Most problems as stated are symptoms, not causes.

### Method:

1. Take the stated problem.
2. Ask "why is this happening?"
3. Take the answer. Ask "why is THAT happening?"
4. Repeat until you hit a statement that feels like a design choice someone made (or forgot to make) rather than a result someone is observing.

### When to stop:

When the answer sounds like a decision rather than an observation. "Nobody designed X" is a decision. "Retention is low" is an observation. Solve decisions, not observations.

---

## Check 2: The Audience Test

**Ask: "Whose problem is this, really?"**

Problems are often stated from the perspective of the person describing them, not the person experiencing them.

### Method:

1. Take the stated problem.
2. Ask: "Who actually experiences this as a problem in their life?"
3. If the answer is "us" (the org, team, lab, department), the problem is stated inside-out. Flip it.
4. What does the person on the other side experience? What would they say the problem is?

The describer's problem and the experiencer's problem often require completely different solutions. Solving the experiencer's problem frequently fixes the describer's problem too. The reverse is not reliable.

---

## Check 3: The Verb Test

**Ask: "What if we changed the action?"**

Most problems contain an implicit verb that predetermines the kind of solution you'll generate.

### Method:

1. Take the stated problem.
2. Identify the verb.
3. Replace it with different verbs and see if the problem becomes more interesting:

- Replace "increase" with "eliminate the need for"
- Replace "reduce" with "make irrelevant"
- Replace "improve" with "replace entirely"
- Replace "fix" with "redesign so it can't break"
- Replace "grow" with "deepen"
- Replace "stop" with "redirect"

Each rewrite points at a different root. The original verb predetermines one category of solution. Different verbs open different doors.

---

## Check 4: The Frame Test

**Ask: "What if the opposite is the real problem?"**

Sometimes the thing that looks like the problem is actually the solution, and the thing everyone assumes is fine is actually the problem.

### Method:

1. Take the stated problem.
2. Invert it.

- "We're not growing fast enough" -> "We're not retaining enough to matter"
- "Our product is too expensive" -> "Our product doesn't justify its price"
- "We need more content" -> "We're not getting enough out of the content we have"
- "Nobody knows about us" -> "The people who know about us aren't telling anyone"

The inversion doesn't always work. When it doesn't, the original framing was probably correct. But when the inverted version suddenly feels more true, you've found the wrong problem.

---

## Check 5: The Existence Test

**Ask: "What if this problem doesn't need to exist at all?"**

Some problems only exist because of a system, structure, or assumption that could be removed entirely.

### Method:

1. Take the stated problem.
2. Ask: "What system, rule, or assumption creates this problem?"
3. Ask: "What if that system, rule, or assumption didn't exist?"
4. If the problem disappears when you remove the assumption, the assumption is the real problem.

---

## After the Checks

### If no check fires:

Say so. "I ran five checks and the problem as stated holds up. It's the right problem to solve." Confidence that you're solving the right thing is valuable. Proceed to generation.

### If any check produced a reframe:

Present a clear checkpoint. Lead with the reframe, not the analysis.

### Checkpoint structure:

1. **Stated problem:** [original]
2. **Reframed:** [strongest reframe -- 1-2 sentences max]
3. **Alternative angles:** 1-2 additional reframes, each one sentence
4. **Ask explicitly:** "Which reframe should I generate ideas from? Or if the original problem is actually right, say so and I'll generate from that instead."

### Critical rule -- reframes are problems, not ideas:

Every reframe must be a problem restatement, not a solution direction.

**Test:** Does this reframe lead to MANY DIFFERENT possible solutions? -> It's a reframe.
Does it point toward ONE specific solution? -> It's an idea disguised as a reframe. Rewrite it.

If the user could respond with "yes, let's do that" -- it's an idea.
If the user responds with "yes, that's the real question" -- it's a reframe.

Wait for the user's response before generating.

---

## Quality Criteria

1. All five checks are run, even when the problem seems obviously correct.
2. The symptom test digs at least 3 levels deep (3 "why" iterations minimum).
3. The audience test identifies the experiencer, not just the describer.
4. The verb test produces at least 2-3 alternative verb framings.
5. Reframes are problems (leading to many solutions), not ideas (pointing to one solution).
6. The checkpoint is impossible to miss -- it leads the response, not buried at the bottom.
7. The user chooses the path before generation begins.

---

## Self-Check

1. Did you actually run all five checks, or did you stop after the first one that fired? Run all five -- the one you'd skip is often the most revealing.
2. In the Symptom Test, did you stop asking "why" too early? If the final answer still sounds like something you'd observe rather than something someone decided, go deeper.
3. In the Audience Test, is the "other side" a real person with a real experience, or a generic label? Make them specific.
4. Are your reframes actually problems? Apply the test: does each lead to many different solutions, or does it point toward one? If one, rewrite.
5. Is the checkpoint presented before any generation begins? The user must choose the problem before you solve it.
